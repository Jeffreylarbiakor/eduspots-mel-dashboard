# EduSpots — Spot Health & MEL Dashboard

A prototype network-monitoring dashboard for EduSpots' community-led learning hubs, built as a working example for the **Head of Programme and Product Operations** application.

**[View the live demo](https://jeffreylarbiakor.github.io/eduspots-mel-dashboard/)**

## Why this exists

The JD names this as a 2026 priority:

> *"As EduSpots advances its monitoring, evaluation and learning strategy in 2026, the Head of Programme and Product Operations will play a key role in strengthening data systems, impact measurement, and organisational learning, overseeing network-wide data collection and analysis."*

and, under Spot development:

> *"Leading on tracking of Spot outcomes and development, working alongside the MEL team to use data effectively to inform the wider team's strategic decision-making, and enabling communities to use data effectively to inform and feed into local activity."*

This dashboard is one concrete answer to "what would that look like in practice" — a single view that turns Spot-level activity data into something an RC or the leadership team can act on, without waiting for a quarterly visit to find out a Spot has gone quiet.

## What it does

- **Overview** — network-wide KPIs (active Spots, Catalysts, Sparks reached, safeguarding compliance) and 12-week engagement trend and strand participation charts.
- **Spot health grid** — every Spot in the network as a status tile (Healthy / Watch / At risk), filterable by status and by RC cluster.
- **RC clusters** — status breakdown per Regional Coordinator, so cluster-level patterns are visible at a glance.
- **Safeguarding** — Spots below the 95% training-compliance target, sorted by urgency.
- **Anomalies** — an auto-generated, RC-facing worklist of flagged Spots with the *specific reason* each was flagged, and an "Acknowledge for follow-up" action.

## Design decision: automation flags, humans decide

This is deliberate, not a missing feature. Every flag on this dashboard is generated by simple, transparent threshold rules (visible in `js/data.js`, function `computeStatus`) — no opaque scoring, and no auto-resolution of anything. The **Acknowledge** button in the Anomalies view exists to make that split visible: the system's job stops at surfacing "here's what needs a human," never at deciding what happens next. This mirrors the AI/human split I laid out for safeguarding and MEL specifically — flagging is automatable, judgment about a child, a Catalyst, or a community isn't.

## Data

All data is **synthetic**, generated client-side by a seeded random function (`js/data.js`), but the shape is modelled on EduSpots' real structure: Spots, Regional Coordinators and their clusters, Catalyst counts, the four learner strands (EduKidz, DigiLit, EcoSTEM, Ignite Equity), and safeguarding training compliance. No real EduSpots data, systems, or credentials were used or accessed in building this.

In production, `generateNetworkData()` in `js/data.js` is the one function that would be replaced with a real fetch to the EduSpots App's data export/API — nothing else in the app would need to change, since every view reads from the same data shape.

## Stack

Deliberately dependency-free: plain HTML/CSS/JS plus [Chart.js](https://www.chartjs.org/) via CDN. No build step, no npm install required to run it — this was a conscious choice so it's trivial to review, fork, or hand off, and so it deploys to GitHub Pages with zero configuration.

## Running it locally

```bash
git clone <your-repo-url>
cd eduspots-mel-dashboard
python3 -m http.server 8000
# then open http://localhost:8000
```

Or just open `index.html` directly in a browser.

## Deploying (GitHub Pages)

1. Push this repo to GitHub.
2. Repo **Settings → Pages → Source**: deploy from `main` branch, root folder.
3. Your live demo URL will be `https://<username>.github.io/<repo-name>/`.

## Roadmap (what I'd build next)

- Swap synthetic data for a real EduSpots App export (CSV/API) — architecture already supports this as a one-function change.
- Add a Catalyst-level (not just Spot-level) safeguarding training view, since compliance is ultimately tracked per person.
- Role-based views: an RC should see only their cluster by default.
- Persist "Acknowledged" state server-side rather than in-memory, and log it as an audit trail for MEL reporting.
- Wire the "Simulate app sync" button to a real polling interval once there's a live data source.
